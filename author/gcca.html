Notas sobre Feature Flags

El Auge y la Complejidad de los Feature Flags en DevOps

En la última década, los feature flags se han convertido en una herramienta fundamental dentro del ecosistema de despliegue, DevOps y mantenimiento, especialmente para la implementación continua. Permitir que operaciones e ingeniería operen de forma independiente ha sido una de sus mayores ventajas, pero no viene sin desafíos.

Ventajas y Desafíos:

  • Ventajas: Los feature flags permiten lanzamientos controlados, experimentación y la capacidad de activar o desactivar características sin necesidad de nuevos despliegues. Esto facilita el despliegue continuo y la innovación ágil.

  • Desafíos: Sin embargo, añaden complexity al código, aumentan el overhead de pruebas, y pueden llevar a bugs si no se gestionan adecuadamente. La lack de procesos automatizados para el seguimiento y la limpieza de feature flags puede resultar en:

  • Zombie flags: Feature flags olvidados que siguen en el código, incrementando el carrying cost y afectando la legibilidad y mantenimiento del código.

  • Unfinished business: Despliegues incompletos de feature flags, causando problemas a largo plazo.

  • Ghost flags: Feature flags eliminados pero aún manipulados por usuarios sin saberlo.

Uso Inapropiado:

  • Un feature flag pensado para un lanzamiento puede usarse para control de acceso a largo plazo, lo que puede llevar a problemas si el servicio externo falla. Esto ha llevado a algunos equipos a separar los sistemas de feature flags en:

  • Short-lived flags para lanzamientos temporales.

  • Long-lived flags para controles de acceso o características permanentes.

Esta separación, aunque lógica, trae consigo:

  • Lack of portability: Dificultad para mover un flag de un sistema a otro sin cambios de código.
  • User confusion: Frustración por recordar en qué sistema está cada flag.
  • Lack of context: Nombres de flags que no explican su propósito sin documentación adicional.

La Realidad de la Gestión de Feature Flags:

El sentido común sugiere que los feature flags deben ser temporales y limitados, pero la realidad del desarrollo de software moderno, con su ritmo rápido y naturaleza multifuncional, hace esto difícil de mantener. Además, la feature management va más allá del ámbito de DevOps, abarcando también la gestión de características para usuarios en diferentes contextos.

Propuestas para Mejorar:

  • Herramientas de Gestión de Feature Flags: Necesitamos herramientas que puedan manejar tanto short-lived como long-lived flags de manera cohesiva, con funciones como:

  • Metadatos adicionales para flags de corta duración, incluyendo fechas de expiración.

  • Asignación de propietarios para cada flag.
  • Capacidad de cambiar el propósito de un flag dentro de la misma herramienta.

  • Políticas automatizadas para la gestión y revisión de flags.

  • Automatización: Implementar procesos automatizados de auditoría y mantenimiento para evitar el crecimiento descontrolado de la complejidad.

Conclusión:

Necesitamos redefinir qué significa la feature management en la industria, alineando las prácticas y herramientas con la realidad operativa de nuestros negocios. Esto podría incluir la creación de herramientas que manejen la complejidad inherente de los feature flags, mejorando así la eficiencia y reduciendo los riesgos asociados con su gestión.

GitHub Actions en monorepos complejos

Resumen del artículo sobre GitHub Actions

  • Contexto: Un equipo trabaja en un monorepo con múltiples módulos, cada uno con su propio pipeline. Desarrollo basado en trunk y múltiples despliegues diarios.

  • Limitaciones de GitHub Actions:

  • Manejo de Workflows Complejos: Problemas con workflows complejos en monorepos, especialmente en la reutilización de workflows y en las comprobaciones de pull request.

  • Dependencia en el Entorno de GitHub: Limitaciones para correr localmente los mismos workflows de GitHub, afectando desarrollo y depuración.

  • Alternativas y Sugerencias:

  • Uso de herramientas como Nix o Docker para ejecutar comandos, permitiendo consistencia entre entornos locales y de CI/CD.

  • Conclusión: Para proyectos grandes y complejos, especialmente monorepos, se recomienda evaluar cuidadosamente el uso de GitHub Actions debido a sus limitaciones en gestión de workflows y consistencia de entornos.

Fast Inverse Square Root (Quake III Arena)

Cada milisegundo cuenta, especialmente en juegos en tiempo real como los shooters en primera persona. Una de las técnicas de optimización más legendarias y controvertidas es el "Fast Inverse Square Root", implementada en el motor de Quake III Arena.

¿Qué es?

El "Fast Inverse Square Root" es un algoritmo diseñado para calcular la raíz cuadrada inversa de un número flotante rápidamente, mucho más rápido que los métodos tradicionales. En términos matemáticos, si tenemos un número x, queremos calcular 1/sqrt(x) de una manera eficiente.

Quake III Arena

El código que nos ha dejado perplejos a muchos es el siguiente:

float Q_rsqrt( float number )
{
    long i;
    float x2, y;
    const float threehalfs = 1.5F;

    x2 = number * 0.5F;
    y  = number;
    i  = * ( long * ) &y;                       // evil floating point bit level hacking
    i  = 0x5f3759df - ( i >> 1 );               // what the fuck?
    y  = * ( float * ) &i;
    y  = y * ( threehalfs - ( x2 * y * y ) );   // 1st iteration
//  y  = y * ( threehalfs - ( x2 * y * y ) );   // 2nd iteration, this can be removed

    return y;
}

Desglose

1 Conversión de bits:

  • i = * (long *) &y; Aquí, el código realiza un "hack" a nivel de bits. La representación en memoria de un número flotante en IEEE 754 es convertida a un entero largo. Esto permite manipular directamente los bits del número flotante.

2 La aproximación:

  • i = 0x5f3759df - (i >> 1); La constante 0x5f3759df es crucial. Este número mágico fue encontrado mediante experimentación y proporciona una primera aproximación sorprendentemente precisa de 1/sqrt(x). La operación i >> 1 es un desplazamiento de bits a la derecha, efectivamente dividiendo i por 2.

3 Refinamiento:

  • y = * (float *) &i; Convierte de vuelta el entero largo a un flotante.
  • y = y * (threehalfs - (x2 * y * y)); Aplica una iteración de Newton-Raphson para mejorar la precisión de la aproximación inicial. La constante threehalfs (1.5 en forma flotante) es usada en este paso de refinamiento.

¿Por qué funciona?

Esta técnica se aprovecha de las propiedades de la representación de punto flotante. La relación entre el logaritmo de un número y su inversa raíz cuadrada es casi lineal cuando se trabaja con los exponentes y mantisas de los números flotantes. La constante 0x5f3759df está ajustada para esta relación, proporcionando una aproximación inicial que luego se refina.

Implicaciones y uso

El "Fast Inverse Square Root" fue fundamental en Quake III porque era usado en cálculos de vectores y normalización, operaciones comunes en gráficos 3D que necesitan ser extremadamente rápidas. Aunque hoy en día los procesadores son mucho más potentes, hay instrucciones específicas para cálculos de punto flotante y tenemos GPUs, este truco histórico sigue siendo un ejemplo impresionante de ingenio.
